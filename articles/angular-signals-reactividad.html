Angular Signals: el futuro de la reactividad

TL;DR: Los Signals son una API reactiva en Angular con estado observable síncrono y dependencias rastreadas. Se crean con signal(), se derivan con computed() y reaccionan con effect(). Traen rendimiento predecible, menor complejidad que cadenas de RxJS para estado local y abren la puerta a zoneless y nuevas optimizaciones del framework.

¿Qué problema resuelven los Signals?

Para estado local de componentes, RxJS suele ser demasiado potente/verboso. Los Signals ofrecen un modelo más simple: el valor se lee como una función (p. ej. count()), Angular rastrea automáticamente qué depende de qué y vuelve a renderizar solo lo necesario.

  • Síncronos y pull-based: lees el valor cuando lo necesitas (count()).
  • Deterministas: no hay multicasting, suscripciones ocultas ni tearing.
  • Declarativos: dependencias inferidas, sin wiring manual.

API básica

import { Component, signal, computed, effect } from '@angular/core';

@Component({
  standalone: true,
  selector: 'demo-counter',
  template: `
    <div class="card">
      <p>Contador: {{ count() }}</p>
      <p>Es par: {{ isEven() ? 'sí' : 'no' }}</p>
      <button (click)="inc()">+1</button>
      <button (click)="reset()">reset</button>
    </div>
  `
})
export class DemoCounter {
  // 1) estado
  readonly count = signal(0);

  // 2) derivado puro
  readonly isEven = computed(() => this.count() % 2 === 0);

  // 3) efecto (side-effects, logging, analytics, etc.)
  private log = effect(() => {
    console.log('nuevo valor', this.count());
  });

  inc()   { this.count.update(v => v + 1); }
  reset() { this.count.set(0); }
}

Mutación:

  • set(v): asigna un valor nuevo.
  • update(fn): calcula a partir del actual.
  • mutate(fn): para mutar estructuras (arrays/objetos) sin recrearlas.
const items = signal<string[]>([]);
items.mutate(arr => arr.push('nuevo')); // no recrea el array

Derivados con computed()

Los computed son funciones puras memorizadas automáticamente. Se recalculan solo cuando cambian sus dependencias.

const price = signal(100);
const tax   = signal(0.19);
const total = computed(() => price() * (1 + tax()));

Efectos con effect() (y limpieza)

Usa effect() para side-effects: escribir en localStorage, disparar analytics, timers, etc. Maneja la limpieza con el token onCleanup().

import { effect, onCleanup } from '@angular/core';

const tick = signal(0);

const timer = effect(() => {
  const id = setInterval(() => tick.update(v => v + 1), 1000);
  onCleanup(() => clearInterval(id));
});

Interoperabilidad con RxJS

Signals y RxJS coexisten. Para integrarte con librerías que exponen Observables usa el paquete de interop.

import { toSignal, toObservable } from '@angular/core/rxjs-interop';
import { of, map } from 'rxjs';

// Observable -> Signal
const user$ = of({ name: 'Ada' });
const user  = toSignal(user$, { initialValue: { name: '' } });

// Signal -> Observable
const name$ = toObservable(user).pipe(map(u => u.name));

Regla práctica: RxJS para flujos asincrónicos/streams (HTTP, websockets, eventos), Signals para estado local y derivaciones de UI.

Inputs como Signals (Angular 17+)

Puedes declarar entradas del componente como signals con input() (y derivarlas con computed()).

import { Component, input, computed } from '@angular/core';

@Component({
  standalone: true,
  selector: 'product-price',
  template: `
    <p>Precio: {{ price() | currency:'USD' }}</p>
    <p>Precio con IVA: {{ withTax() | currency:'USD' }}</p>
  `
})
export class ProductPrice {
  price = input.required<number>();
  withTax = computed(() => this.price() * 1.19);
}

Patrones útiles

1) Estado de formulario sencillo

import { Component, signal, computed } from '@angular/core';

@Component({
  standalone: true,
  selector: 'contact-form',
  template: `
    <form (ngSubmit)="submit()">
      <input [(ngModel)]="nameLocal" name="name" placeholder="Nombre">
      <textarea [(ngModel)]="msgLocal" name="msg" placeholder="Mensaje"></textarea>

      <button [disabled]="!canSend()">Enviar</button>
    </form>
  `
})
export class ContactForm {
  private name  = signal('');
  private msg   = signal('');
  // two–way binding adaptado
  get nameLocal() { return this.name(); }
  set nameLocal(v: string) { this.name.set(v); }
  get msgLocal() { return this.msg(); }
  set msgLocal(v: string) { this.msg.set(v); }

  canSend = computed(() => this.name().length >= 3 && this.msg().length >= 10);

  submit() { /* ... */ }
}

2) Store ligera con servicio inyectable

import { Injectable, signal, computed } from '@angular/core';

export interface Todo { id: number; text: string; done: boolean; }

@Injectable({ providedIn: 'root' })
export class TodoStore {
  private list = signal<Todo[]>([]);
  readonly todos = computed(() => this.list());
  readonly pending = computed(() => this.list().filter(t => !t.done));
  readonly done = computed(() => this.list().filter(t =>  t.done));

  add(text: string) {
    this.list.update(xs => [...xs, { id: Date.now(), text, done: false }]);
  }
  toggle(id: number) {
    this.list.mutate(xs => {
      const t = xs.find(x => x.id === id);
      if (t) t.done = !t.done;
    });
  }
}

Rendimiento y change detection

  • Los Signals permiten que Angular reprocese solo lo que depende del valor cambiado.
  • Funcionan muy bien con componentes OnPush.
  • Facilitan optimizaciones hacia un futuro zone-less (sin Zone.js) y renderizado más fino.

Buenas prácticas

  • Usa computed() para derivaciones puras. No pongas side-effects dentro.
  • Usa effect() solo para efectos secundarios (I/O, logs, timers). Recuerda onCleanup().
  • No mutes objetos/arrays sin mutate(). O recrea con update().
  • RxJS sigue vivo para flujos asincrónicos; no intentes reescribir todo si ya funciona bien.

Errores comunes

  • Leer un signal sin invocarlo (debes usar count(), no count).
  • Olvidar limpiar recursos en effect() con onCleanup().
  • Hacer trabajo pesado dentro de effect() en cada cambio: llévalo a un computed() y memoriza.

Checklist para migrar paso a paso

  • ✅ Mantén RxJS en servicios/HTTP; usa Signals para estado local y derivado de UI.
  • ✅ Convierte un BehaviorSubject a signal() solo si se simplifica.
  • ✅ Usa toSignal() / toObservable() para interop puntual.
  • ✅ Encapsula lógica compartida en stores inyectables con signals.

Conclusión: Signals hacen que el estado de UI en Angular sea más claro y predecible. Empieza por componentes con estado local, deriva valores con computed() y reserva RxJS para lo que mejor hace: flujos asíncronos y composición avanzada.
