Implementando Clean Architecture en .NET

TL;DR: separa tu código en capas con dependencias hacia adentro (Domain → Application → Infrastructure → Web). Define reglas del negocio en Domain, casos de uso en Application, detalles técnicos en Infrastructure y exposición HTTP en Web/API. Así logras testabilidad, menor acoplamiento y un sistema fácil de evolucionar.

¿Qué es Clean Architecture y por qué usarla?

Clean Architecture propone organizar el software en capas concéntricas, donde las dependencias apuntan al núcleo del negocio. La idea: tu lógica de negocio debe sobrevivir aunque cambies la base de datos, el framework web o el proveedor de nube.

┌───────────────────────────┐   Web / API (endpoints, controllers, DTOs)
│           Web             │
├───────────────────────────┤   Infrastructure (EF Core, Stripe, Redis, Email…)
│      Infrastructure       │
├───────────────────────────┤   Application (casos de uso, orquestación)
│        Application        │
├───────────────────────────┤   Domain (entidades, value objects, reglas)
│          Domain           │
└───────────────────────────┘
  • Domain: Entidades, Value Objects, eventos de dominio y reglas puras (sin depender de .NET ni EF).
  • Application: Casos de uso (comandos/consultas), puertos (interfaces), validaciones y orquestación.
  • Infrastructure: Implementaciones técnicas de los puertos: EF Core, HTTP, colas, archivos, etc.
  • Web/API: Endpoints, serialización/deserialización, autenticación/autorización y filtros.

Estructura mínima de carpetas

src/
  Domain/
    Entities/
    ValueObjects/
    Abstractions/
  Application/
    UseCases/
    Abstractions/

1) Capa Domain: entidades y value objects

Define tu modelo de negocio sin dependencias. Ejemplo: un Product con un Money como value object.

namespace Domain.ValueObjects;

public sealed record Money(decimal Amount, string Currency)
{
    public static Money From(decimal amount, string currency = "USD")
    {
        if (amount < 0) throw new ArgumentException("Amount must be positive.");
        if (string.IsNullOrWhiteSpace(currency)) throw new ArgumentException("Currency required.");
        return new Money(amount, currency.ToUpperInvariant());
    }

    public Money Add(Money other)
        => Currency != other.Currency
            ? throw new InvalidOperationException("Different currencies.")
            : this with { Amount = Amount + other.Amount };
}

namespace Domain.Entities;

using Domain.ValueObjects;

public sealed class Product
{
    private Product() { } // EF
    public Guid Id { get; private set; } = Guid.NewGuid();
    public string Name { get; private set; } = default!;
    public Money Price { get; private set; } = Money.From(0);

    public Product(string name, Money price)
    {
        Rename(name);
        ChangePrice(price);
    }

    public void Rename(string name)
    {
        if (string.IsNullOrWhiteSpace(name)) throw new ArgumentException("Name required.");
        Name = name.Trim();
    }

    public void ChangePrice(Money price)
    {
        if (price.Amount == 0) throw new ArgumentException("Price must be > 0.");
        Price = price;
    }
}

2) Capa Application: casos de uso y puertos

La aplicación orquesta la lógica llamando a puertos (interfaces) que luego Infrastructure implementa. Puedes usar MediatR, pero aquí lo mostramos simple para que sea claro.

namespace Application.Abstractions;

using Domain.Entities;

public interface IProductRepository
{
    Task<Product?> GetByIdAsync(Guid id, CancellationToken ct);
    Task AddAsync(Product product, CancellationToken ct);
    Task SaveChangesAsync(CancellationToken ct);
}

namespace Application.UseCases.Products;

using Application.Abstractions;
using Domain.Entities;
using Domain.ValueObjects;

public sealed class CreateProductCommand
{
    public required string Name { get; init; }
    public required decimal Price { get; init; }
    public string Currency { get; init; } = "USD";
}

public sealed class CreateProductHandler
{
    private readonly IProductRepository _repo;

    public CreateProductHandler(IProductRepository repo) => _repo = repo;

    public async Task<Guid> Handle(CreateProductCommand cmd, CancellationToken ct)
    {
        var product = new Product(cmd.Name, Money.From(cmd.Price, cmd.Currency));
        await _repo.AddAsync(product, ct);
        await _repo.SaveChangesAsync(ct);
        return product.Id;
    }
}

Validación (ejemplo con FluentValidation):

using FluentValidation;

namespace Application.UseCases.Products;

public sealed class CreateProductValidator : AbstractValidator<CreateProductCommand>
{
    public CreateProductValidator()
    {
        RuleFor(x => x.Name).NotEmpty().MaximumLength(120);
        RuleFor(x => x.Price).GreaterThan(0);
        RuleFor(x => x.Currency).NotEmpty().Length(3);
    }
}

3) Capa Infrastructure: EF Core y repositorio

using Domain.Entities;
using Domain.ValueObjects;
using Microsoft.EntityFrameworkCore;

namespace Infrastructure.Persistence;

public sealed class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Product> Products => Set<Product>();

    protected override void OnModelCreating(ModelBuilder model)
    {
        model.Entity<Product>(builder =>
        {
            builder.HasKey(p => p.Id);
            builder.Property(p => p.Name).HasMaxLength(120).IsRequired();

            builder.OwnsOne(p => p.Price, price =>
            {
                price.Property(m => m.Amount).HasColumnType("decimal(18,2)");
                price.Property(m => m.Currency).HasMaxLength(3);
            });
        });
    }
}

using Application.Abstractions;
using Domain.Entities;
using Infrastructure.Persistence;
using Microsoft.EntityFrameworkCore;

namespace Infrastructure.Repositories;

public sealed class ProductRepository : IProductRepository
{
    private readonly AppDbContext _db;
    public ProductRepository(AppDbContext db) => _db = db;

    public async Task<Product?> GetByIdAsync(Guid id, CancellationToken ct)
        => await _db.Products.FirstOrDefaultAsync(p => p.Id == id, ct);

    public async Task AddAsync(Product product, CancellationToken ct)
        => await _db.Products.AddAsync(product, ct);

    public Task SaveChangesAsync(CancellationToken ct)
        => _db.SaveChangesAsync(ct);
}

4) Capa Web/API: Minimal APIs en .NET 8

Expón DTOs de entrada/salida, injerta el handler y valida. Con Minimal APIs la configuración queda compacta.

// Web/Program.cs
using Application.Abstractions;
using Application.UseCases.Products;
using FluentValidation;
using Infrastructure.Persistence;
using Infrastructure.Repositories;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

// Infra
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

// DI
builder.Services.AddScoped<IProductRepository, ProductRepository>();
builder.Services.AddScoped<CreateProductHandler>();
builder.Services.AddScoped<IValidator<CreateProductCommand>, CreateProductValidator>();

var app = builder.Build();

app.MapPost("/products", async (
    CreateProductCommand cmd,
    CreateProductHandler handler,
    IValidator<CreateProductCommand> validator,
    CancellationToken ct) =>
{
    var result = await validator.ValidateAsync(cmd, ct);
    if (!result.IsValid)
        return Results.ValidationProblem(result.ToDictionary());

    var id = await handler.Handle(cmd, ct);
    return Results.Created($"/products/{id}", new { id });
});

app.MapGet("/products/{id:guid}", async (
    Guid id, IProductRepository repo, CancellationToken ct) =>
{
    var product = await repo.GetByIdAsync(id, ct);
    return product is null ? Results.NotFound() : Results.Ok(product);
});

app.Run();

Pruebas unitarias del caso de uso (sin base de datos)

Gracias a los puertos, puedes probar la aplicación con un repositorio en memoria.

using Application.Abstractions;
using Application.UseCases.Products;
using Domain.Entities;
using Domain.ValueObjects;

public sealed class InMemoryProductRepository : IProductRepository
{
    private readonly Dictionary<Guid, Product> _store = new();

    public Task<Product?> GetByIdAsync(Guid id, CancellationToken ct)
        => Task.FromResult(_store.TryGetValue(id, out var p) ? p : null);

    public Task AddAsync(Product product, CancellationToken ct)
    {
        _store[product.Id] = product;
        return Task.CompletedTask;
    }

    public Task SaveChangesAsync(CancellationToken ct) => Task.CompletedTask;
}

// xUnit
public class CreateProductTests
{
    [Fact]
    public async Task Creates_product_with_valid_data()
    {
        var repo = new InMemoryProductRepository();
        var handler = new CreateProductHandler(repo);

        var cmd = new CreateProductCommand { Name = "Cepillo dental", Price = 12.5m, Currency = "USD" };
        var id = await handler.Handle(cmd, default);

        var saved = await repo.GetByIdAsync(id, default);
        Assert.NotNull(saved);
        Assert.Equal("Cepillo dental", saved!.Name);
        Assert.Equal(Money.From(12.5m, "USD"), saved.Price);
    }
}

Migraciones y despliegue

  1. Configura tu cadena de conexión Default en appsettings.json.
  2. Desde el proyecto Web (o donde esté el DbContext), ejecuta: 
    dotnet ef migrations add Initial --project ../Infrastructure --startup-project ./Web
dotnet ef database update --project ../Infrastructure --startup-project ./Web
  3. Despliega. Cambiar SQL Server por Postgres o Azure SQL sólo toca Infrastructure.

Checklist de adopción

  • ✅ Entidades y Value Objects en Domain sin dependencias.
  • ✅ Casos de uso orquestan lógica en Application y llaman a puertos.
  • ✅ EF Core, repos, integraciones en Infrastructure.
  • ✅ Endpoints/DTOs y auth en Web.
  • ✅ Validaciones con FluentValidation y pruebas de casos de uso con dobles en memoria.
  • ✅ Sin acoplar el dominio a EF, HTTP o librerías externas.

Variantes y recomendaciones

  • CQRS + MediatR: si el proyecto crece, separa comandos/consultas y usa pipelines (logging, validación, transacciones).
  • Mapeo: Mapster o AutoMapper para transformar DTOs <-> domain.
  • Transacciones: maneja Unit of Work en Infrastructure (con el propio DbContext) o filtros en Web.
  • Eventos de Dominio: emite cambios importantes y manéjalos en Application para side-effects (emails, auditoría, etc.).

Conclusión: Clean Architecture no es una religión, es una guía. Empieza simple (como arriba) y agrega complejidad sólo cuando tu producto lo pida. Así mantienes tu base de código clara, testeable y lista para crecer.
