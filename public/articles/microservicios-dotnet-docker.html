Microservicios con .NET y Docker

TL;DR: Divide tu solución en servicios pequeños y autónomos (bounded contexts) con base de datos por servicio, comunícalos por HTTP/gRPC y mensajería (RabbitMQ), publícalos como contenedores y orquéstralos con docker-compose (o Kubernetes). Gana despliegues independientes, escalado selectivo y resiliencia. Usa OpenTelemetry para observabilidad y un API Gateway para enrutar y proteger.

¿Cuándo (no) usar microservicios?

  • Úsalos si necesitas equipos autónomos, despliegues independientes, escalado desigual (por ejemplo, Orders escala mucho más que Catalog), o si el dominio es amplio con límites claros.
  • No los uses aún si tu producto es pequeño, el equipo es de 1–3 personas o no tienes capacidad para monitoreo, CI/CD y DevOps. Empieza como monolito modular y extrae servicios cuando el dolor lo justifique.

1) Catálogo: Minimal API en .NET 8

Un endpoint de ejemplo. Observa que el dominio no conoce EF ni HTTP.

// Catalog.Domain/Entities/Product.cs
namespace Catalog.Domain.Entities;

public sealed class Product
{
    public Guid Id { get; private set; } = Guid.NewGuid();
    public string Name { get; private set; } = string.Empty;
    public decimal Price { get; private set; }

    public Product(string name, decimal price)
    {
        if (string.IsNullOrWhiteSpace(name)) throw new ArgumentException("Name required.");
        if (price <= 0) throw new ArgumentException("Price must be > 0");
        Name = name.Trim();
        Price = price;
    }
}

// Catalog.Infrastructure/Persistence/AppDbContext.cs
using Catalog.Domain.Entities;
using Microsoft.EntityFrameworkCore;

namespace Catalog.Infrastructure.Persistence;

public sealed class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> opt) : base(opt) { }
    public DbSet<Product> Products => Set<Product>();

    protected override void OnModelCreating(ModelBuilder model)
    {
        model.Entity<Product>(b => {
            b.HasKey(p => p.Id);
            b.Property(p => p.Name).HasMaxLength(120).IsRequired();
            b.Property(p => p.Price).HasColumnType("numeric(18,2)");
        });
    }
}

// Catalog.Api/Program.cs
using Catalog.Infrastructure.Persistence;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseNpgsql(builder.Configuration.GetConnectionString("CatalogDb")));

builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

app.MapGet("/products", async (AppDbContext db) =>
    await db.Products.AsNoTracking().ToListAsync());

app.MapPost("/products", async (AppDbContext db, ProductDto dto) =>
{
    var p = new Catalog.Domain.Entities.Product(dto.Name, dto.Price);
    db.Add(p);
    await db.SaveChangesAsync();
    return Results.Created($"/products/{p.Id}", new { p.Id });
});

app.UseSwagger();
app.UseSwaggerUI();
app.Run();

record ProductDto(string Name, decimal Price);

2) Dockerfile para .NET 8

# Catalog.Api/Dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
WORKDIR /app
EXPOSE 8080

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ./Catalog.Api/*.csproj ./Catalog.Api/
COPY ./Catalog.Infrastructure/*.csproj ./Catalog.Infrastructure/
COPY ./Catalog.Domain/*.csproj ./Catalog.Domain/
RUN dotnet restore ./Catalog.Api/Catalog.Api.csproj

COPY . .
RUN dotnet publish ./Catalog.Api/Catalog.Api.csproj -c Release -o /out /p:UseAppHost=false

FROM base AS final
WORKDIR /app
COPY --from=build /out .
ENTRYPOINT ["dotnet", "Catalog.Api.dll"]

3) docker-compose para levantar todo

Incluye PostgreSQL, RabbitMQ, Jaeger (tracing) y un API Gateway con YARP.

# docker/docker-compose.yml
services:
  catalog.api:
    build: ../src/services/catalog/Catalog.Api
    container_name: catalog.api
    environment:
      - ASPNETCORE_URLS=http://+:8080
      - ConnectionStrings__CatalogDb=Host=postgres;Database=catalog;Username=postgres;Password=postgres
    depends_on: [postgres]
    networks: [appnet]
    ports: ["8081:8080"]

  orders.api:
    build: ../src/services/orders/Orders.Api
    container_name: orders.api
    environment:
      - ASPNETCORE_URLS=http://+:8080
      - ConnectionStrings__OrdersDb=Host=postgres;Database=orders;Username=postgres;Password=postgres
      - MessageBroker__Host=rabbitmq
    depends_on: [postgres, rabbitmq]
    networks: [appnet]
    ports: ["8082:8080"]

  gateway:
    build: ../src/gateway/Yarp.Gateway
    container_name: gateway
    environment:
      - ASPNETCORE_URLS=http://+:8080
    depends_on: [catalog.api, orders.api]
    networks: [appnet]
    ports: ["8080:8080"]

  postgres:
    image: postgres:16-alpine
    container_name: postgres
    environment:
      - POSTGRES_PASSWORD=postgres
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks: [appnet]
    ports: ["5432:5432"]

  rabbitmq:
    image: rabbitmq:3-management-alpine
    container_name: rabbitmq
    networks: [appnet]
    ports:
      - "5672:5672"
      - "15672:15672"   # consola web

  jaeger:
    image: jaegertracing/all-in-one:1.57
    container_name: jaeger
    networks: [appnet]
    ports:
      - "16686:16686"   # UI

volumes:
  pgdata:

networks:
  appnet:

Comandos:

docker compose -f docker/docker-compose.yml up --build

4) API Gateway con YARP

{
  "ReverseProxy": {
    "Routes": [
      { "RouteId": "catalog", "ClusterId": "catalog", "Match": { "Path": "/catalog/{**catch-all}" } },
      { "RouteId": "orders",  "ClusterId": "orders",  "Match": { "Path": "/orders/{**catch-all}"  } }
    ],
    "Clusters": {
      "catalog": { "Destinations": { "d1": { "Address": "http://catalog.api:8080" } } },
      "orders":  { "Destinations": { "d1": { "Address": "http://orders.api:8080"  } } }
    }
  }
}

En Program.cs de Gateway:

builder.Services.AddReverseProxy().LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));
app.MapReverseProxy();

5) Comunicación entre servicios

HTTP (sincrónico)

// Orders.Api llama a Catalog.Api para verificar stock/precio
var http = new HttpClient { BaseAddress = new Uri("http://catalog.api:8080") };
var product = await http.GetFromJsonAsync<ProductDto>($"/products/{productId}");

Eventos (asíncrono) con RabbitMQ

Define un contrato y un bus mínimo:

// Messaging.Abstractions/IEventBus.cs
public interface IEventBus {
    Task PublishAsync<T>(T @event, CancellationToken ct = default);
    Task SubscribeAsync<T>(Func<T, Task> handler, CancellationToken ct = default);
}

public record ProductPriceChanged(Guid ProductId, decimal NewPrice);

// Orders.Api: manejador del evento (actualiza precios locales, cache, etc.)
await _eventBus.SubscribeAsync<ProductPriceChanged>(async evt => {
    await _ordersService.UpdatePriceAsync(evt.ProductId, evt.NewPrice);
});

Implementación RabbitMQ: canaliza Publish a un exchange por tipo y Subscribe a una queue por servicio (omito por brevedad, pero la interface anterior te lo deja listo).

8) Seguridad y configuración

  • JWT en Gateway (validación) y propagación de claims a servicios.
  • Usa variables de entorno y Secret Manager para credenciales en desarrollo.
  • Habilita HTTPS en gateway y, si puedes, mTLS entre servicios internos.

9) Despliegue

Publica imágenes en tu registry (Azure Container Registry, GHCR, Docker Hub) y usa Kubernetes para producción:

# k8s/catalog-deployment.yaml (mini ejemplo)
apiVersion: apps/v1
kind: Deployment
metadata: { name: catalog-api }
spec:
  replicas: 2
  selector: { matchLabels: { app: catalog-api } }
  template:
    metadata: { labels: { app: catalog-api } }
    spec:
      containers:
        - name: catalog
          image: your-registry/catalog.api:1.0.0
          ports: [{ containerPort: 8080 }]
          env:
            - name: ConnectionStrings__CatalogDb
              valueFrom:
                secretKeyRef: { name: app-secrets, key: catalogdb }
---
apiVersion: v1
kind: Service
metadata: { name: catalog-api }
spec:
  selector: { app: catalog-api }
  ports: [{ port: 80, targetPort: 8080 }]

Checklist de producción

  • ✅ Base de datos por servicio y límites de dominio claros.
  • ✅ Idempotencia y retries/backoff en consumidores de eventos y HTTP.
  • ✅ Tracing/metrics/logging centralizados (OTel + Jaeger/Prometheus/Seq).
  • ✅ Health checks y readiness/liveness para el orquestador.
  • ✅ CI/CD con build, test y push de imágenes por servicio.
  • ✅ Seguridad: JWT, secretos fuera del repo, HTTPS, mínimos privilegios.

Conclusión: Microservicios con .NET + Docker funcionan muy bien cuando el dominio lo requiere y cuentas con las piezas de observabilidad y automatización. Empieza pequeño con 2–3 servicios y docker-compose, define contratos claros y añade mensajería para desacoplar. Cuando la escala lo pida, pasa a Kubernetes.
